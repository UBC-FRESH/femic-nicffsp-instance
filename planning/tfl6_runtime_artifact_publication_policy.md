# TFL 6 Runtime Artifact Publication Policy

## Purpose

This note closes P5.1 by deciding how TFL 6 runtime-package artifacts are
handled after Phase 4 runtime QA. It does not publish data. It defines what
Phase 5 publication and release QA must prove next.

The Phase 4 runtime package generated these local surfaces:

| Surface | Files | Approx. size | Policy |

| `models/tfl6_patchworks_model/tracks/` | 13 | 25.02 MB | regenerated, optionally packaged in P5.2 |

These sizes are small enough that a future reviewed release archive is
practical, but large and mechanical enough that they should not be committed
directly to Git as ordinary source files.

## Policy

### Tracked In Git

Track compact files that describe the runtime package, its commands, and its
launch surface:

- `models/tfl6_patchworks_model/README.md`;
- `models/tfl6_patchworks_model/lineage_registry.yaml`;
- `models/tfl6_patchworks_model/analysis/base.pin`;
- `models/tfl6_patchworks_model/analysis/base_variant_common.bsh`;
- `models/tfl6_patchworks_model/analysis/headless_runtime_common.bsh`;
- `models/tfl6_patchworks_model/scripts/targets/flowtargets.bsh`;
- instance configs, docs, planning notes, and changelog entries that explain
  how to regenerate or use the package.

These files are human-reviewable contract surfaces. They should remain tracked
in this repository.

### Regenerated By Default

The following surfaces are generated outputs and remain ignored by Git:

- `data/model_input_bundle/`;
- `output/patchworks_tfl6_validated/`;
- `models/tfl6_patchworks_model/tracks/`;
- `models/tfl6_patchworks_model/blocks/`;
- `models/tfl6_patchworks_model/patchworksLog.csv`;
- runtime logs and manifests under `runtime/`.

The accepted rebuild commands are recorded in:

- `models/tfl6_patchworks_model/lineage_registry.yaml`;
- `planning/tfl6_model_input_bundle_core_tables.md`;
- `planning/tfl6_forestmodel_xml_export_bridge.md`;
- `planning/tfl6_matrix_builder_p43_smoke.md`;
- `planning/tfl6_runtime_package_p44.md`.

Phase 5 release QA must rerun or materialize these surfaces before claiming a
fresh environment can build or launch the model.

### Candidate Published Runtime Package

P5.2 may create a reviewed runtime release archive if the teaching workflow
needs a no-rebuild Patchworks launch package. That archive should be a deliberate
release artifact, not a directory dump into Git. The candidate archive should
include:

- `output/patchworks_tfl6_validated/forestmodel.xml`;
- `output/patchworks_tfl6_validated/fragments/fragments.*`;
- `models/tfl6_patchworks_model/tracks/*.csv`;
- `models/tfl6_patchworks_model/blocks/blocks.*`;
- `models/tfl6_patchworks_model/blocks/topology_blocks_200r.csv`;
- the tracked `analysis/*.pin` / `.bsh` launch surface;
- `models/tfl6_patchworks_model/README.md`;
- `models/tfl6_patchworks_model/lineage_registry.yaml`.

If produced, the archive must be annexed/published through the accepted FEMIC
public-data workflow or another explicitly documented public release mechanism.
P5.2 must then prove fresh-environment materialization before release readiness
is claimed.

### Excluded From Publication

Saved-stage smoke output remains excluded from normal publication:

- `models/tfl6_patchworks_model/analysis/p44b_launch0/`;
- `models/tfl6_patchworks_model/analysis/p44d_harvest_smoke200/`;
- other `models/tfl6_patchworks_model/analysis/p44*/` saved stages;
- `models/tfl6_patchworks_model/analysis/headless_runs/`.

These directories are validation evidence, not canonical release inputs. Their
important facts are recorded in planning notes, issue comments, and
`lineage_registry.yaml`.

## P5.2 Handoff

P5.2 should decide whether to:

1. publish only source data plus rebuild instructions, requiring maintainers to
   regenerate the runtime package locally; or
2. additionally publish a reviewed runtime archive for teaching users who need a
   ready-to-launch Patchworks package.

Either path is acceptable only after a fresh-environment materialization check.
Local-only availability is not release evidence.
